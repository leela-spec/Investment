# C5 — Playbook Integration & Retrieval (Meso Plan)

**Owns:** the bridge between the extracted knowledge layer (`03_extract/`, `04_playbook/`) and the runtime: `04_playbook/index.json`, `snapshot_flags.md`, `blueprint_map.md` (all currently **empty placeholders**), deterministic module retrieval for the AI layer.
**Depends on:** C1 (registry `playbook_refs`), C4 (contradictions/movers). **Feeds:** C6.

## Options considered

| Option | Verdict |
|--------|---------|
| Deterministic `playbook_refs` retrieval (Blueprint) | ✅ **Chosen.** Registry lists module IDs per indicator; selection = pure set logic. Zero infra, reproducible, debuggable. |
| Embeddings/RAG | ❌ Confirmed pointless at this scale (10–20 modules, all fit in context); adds infra + nondeterminism for negative value. |
| Feed all modules every run | ❌ ~40k+ tokens/run vs ≤3k budget; violates token design for no quality gain. |

## Decisions

1. **`04_playbook/index.json` (fill the placeholder):** machine-readable catalog generated from module front-matter — `{module_id, path, tags, version, token_estimate, summary_1line}`. Generated by script, never hand-edited; QA fails if a module file lacks front-matter or isn't in the index.
2. **Retrieval rule (deterministic, ordered, budgeted):** include modules referenced by
   1. active contradictions (their `applies_to` modules),
   2. regime block (MARKET_CONDITIONS always; TRAILING_STOP_POLICIES when regime/policy changed),
   3. top movers' `playbook_refs`,
   4. indicators at extremes (score <20 or >80),
   stop when the ≤3,000-token budget is hit (priority order above; `token_estimate` from index). Selection list is itself written into the snapshot (`playbook_selection: [...]`) for auditability.
3. **Module excerpting:** modules are already compact (2–4k chars); v1 includes whole modules. If a module busts the budget (SENTIMENT_POSITIONING, 24k chars), split it at H2 level into sub-excerpts addressed as `MODULE#section` — one-time editorial pass, no tooling.
4. **`snapshot_flags.md` (fill the placeholder):** the canonical dictionary of every boolean/enum flag a module contributes to the snapshot (e.g. `rsi_power_zone_active`, `below_200wma`, `regime_label`, `crv_floor_ok`) — mined from the modules' "Snapshot Requirements (AI + auditability)" sections. Each flag: id, type, producing cluster, meaning. This becomes the contract C3/C4 implement and C6 serializes.
5. **`blueprint_map.md` (fill the placeholder):** one-page traceability table Blueprint-section → cluster (C1–C9) → code path → status. Cheap, keeps plan and code honest; updated per phase.
6. **Provenance discipline:** runtime never reads `03_extract/*.jsonl` (extraction is upstream, ops reads registry/configs) — but every seeded config row keeps `extract_ref` so QA can trace any threshold to its page-ref'd extraction line.

## Implementation steps

1. `scripts/build_playbook_index.py` — front-matter parse → `index.json` (+ token estimate = chars/4); wire into `qa_repo.py`.
2. Author `snapshot_flags.md` from the 10 modules (editorial, ~1 session).
3. Author `blueprint_map.md` skeleton (grows per phase).
4. `ipos/playbook/retrieve.py` — selection algorithm (pure function: snapshot-draft → ordered module list within budget) + unit tests with constructed snapshots.
5. Split SENTIMENT_POSITIONING into H2 sub-excerpts if token_estimate > budget share.

## Definition of done
- Placeholders filled and generated/validated by QA scripts; `retrieve()` returns a stable, budget-conform selection for fixture snapshots; selection recorded in snapshot.

**Effort:** S–M. **Recurring tokens:** 0 (this cluster *saves* ~35k tokens/run vs naive all-modules).
